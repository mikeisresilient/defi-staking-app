# Integration Architecture

This document details the integration architecture between the Frontend (web) and Smart Contracts (backend) components of the DeFi Staking App. As a decentralized application (dApp), the primary integration mechanism is direct interaction with the Ethereum blockchain via Web3.js.

## 1. Integration Points

The Frontend component directly communicates with the Smart Contracts deployed on the Ethereum blockchain.

*   **From**: Frontend (web)
*   **To**: Smart Contracts (backend)
*   **Type**: Web3.js / Ethereum JSON-RPC
*   **Details**:
    *   **Web3 Provider Connection**: The frontend initiates a connection to an Ethereum provider (e.g., MetaMask injected into the browser). This provider acts as the gateway to the blockchain.
    *   **Contract Instantiation**: The frontend uses the Application Binary Interface (ABI) and the deployed address of each smart contract (Tether, RWD, DecentralBank) to create JavaScript instances of these contracts. These ABIs are typically generated by the Truffle framework and stored in `src/truffle_abis/`.
    *   **Function Calls**: The frontend invokes public and external functions of the deployed smart contracts (e.g., `depositTokens`, `unstakeTokens`, `issueTokens`). These calls can be either:
        *   **Read-only calls**: For querying contract state (e.g., `balanceOf`, `stakingBalance`), which do not modify the blockchain state and are typically free.
        *   **Transaction calls**: For functions that modify the blockchain state (e.g., `depositTokens`, `unstakeTokens`), which require a transaction to be signed by the user and submitted to the network, incurring gas fees.
    *   **State Reading**: The frontend reads public state variables and mapping values directly from the smart contracts to display real-time information to the user (e.g., current staking balance, token balances).
    *   **Transaction Submission**: User actions that require blockchain interaction (e.g., staking, unstaking) result in transactions being constructed and sent to the connected Web3 provider for user confirmation and subsequent submission to the Ethereum network.

## 2. Data Flow

*   **Frontend to Blockchain**: User actions (e.g., clicking "Deposit") trigger Web3.js calls that translate into signed Ethereum transactions. These transactions carry data (e.g., amount to stake) to the smart contracts.
*   **Blockchain to Frontend**: The frontend queries the blockchain for contract state (e.g., `balanceOf`, `stakingBalance`) and transaction receipts. Events emitted by smart contracts can also be listened to by the frontend to react to state changes on-chain.

## 3. Authentication Flow

*   **Mechanism**: User authentication is handled by the Web3 provider (e.g., MetaMask). The user grants the dApp permission to access their Ethereum account(s).
*   **Authorization**: Smart contracts enforce authorization rules (e.g., `onlyOwner` modifier for `issueTokens` in `DecentralBank.sol`) based on the `msg.sender` of a transaction.

## 4. Shared Dependencies

*   **Web3.js**: This library is a critical shared dependency, used by both the frontend (for blockchain interaction) and the smart contract development environment (Truffle uses Web3.js internally for deployment and testing).
*   **Smart Contract ABIs**: The JSON files containing the contract ABIs (e.g., `src/truffle_abis/*.json`) are essential for the frontend to correctly encode and decode data when interacting with the deployed contracts.
